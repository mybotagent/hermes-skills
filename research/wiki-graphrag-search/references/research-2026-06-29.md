## 1. OpenKB (VectifyAI) — ⭐2,740

**URL:** https://github.com/VectifyAI/OpenKB
**Stack:** Python, LiteLLM, PageIndex
**설치:** `pip install openkb`

### 핵심 특징
- **CLI 기반** — `openkb init` → `openkb add` → `openkb query` / `openkb chat`
- **PageIndex vectorless retrieval** — 긴 문서(20+ page PDF)를 hierarchical tree index로 변환, vector DB 없이 reasoning-based 검색
- **멀티 포맷** — PDF, Word, Markdown, PPT, HTML, Excel, CSV, URL
- **Native multi-modality** — 표, 이미지, 그림도 이해
- **Generator 3종:**
  - `openkb query` — 일회성 질문
  - `openkb chat` — 대화형
  - `openkb skill new` — 위키에서 portable agent skill 추출 (Skill Factory)
  - `openkb visualize` — interactive knowledge graph (HTML)
  - `openkb deck new` — slide deck 생성 (single-file HTML)
- **OKF (Google Open Knowledge Format)** 호환
- **Obsidian 호환** — wiki는 `.md` plain files

### 우리에 적용할 아이디어
- 문서 → concept + entity 페이지 분리 (우리 frontmatter 시스템 강화)
- Skill Factory: wiki 내용에서 Hermes skill 자동 생성
- visualize: vis-network 그래프 HTML 생성

### Limitation
- CLI 기반이라 Hermes Agent 내부에서 호출하려면 subprocess 필요
- 우리 wiki 구조를 OpenKB 포맷으로 변환 부담

## 2. lcwiki / Karpathy-wiki-graph (LCccode) — ⭐4

**URL:** https://github.com/LCccode/Karpathy-wiki-graph
**Stack:** Python, pypi `lcwiki`, vis-network
**설치:** `pip install lcwiki`

### 핵심 특징
- **3-layer 검색:** articles(100-token tldr) → concepts (standalone) → graph (vis-network)
- **CLI-atomic write-verify** — agent가 KB를 망가뜨릴 수 없음
- **Concept family aliases** — 부서 간 용어 통일 (예: "Digital Platform" / "数字基座")
- **명령어:** `/lcwiki ingest → compile → graph → query`
- **Token 비용:** ~10% of vanilla RAG
- **Incremental training:** ~10× lower ongoing cost
- **구조:**
  ```
  vault/
  ├── wiki/
  │   ├── articles/*.md     per-doc wiki pages with YAML frontmatter + 100-token tldr
  │   └── concepts/*.md     standalone concept pages with family aliases
  ├── graph/
  │   └── graph.html        vis-network interactive knowledge graph
  └── raw/                  inbox for new documents
  ```

### 우리에 적용할 아이디어
- **vis-network 그래프 HTML** — `openkb visualize`와 유사, 우리 wikilink 기반 그래프 생성
- **100-token tldr** — 페이지 요약을 node property로 저장
- **Self-healing audit** — `/lcwiki audit` with LLM-as-judge

### Limitation
- 초기 프로젝트 (⭐4) — 장기 유지보수 불확실
- Claude Code / OpenClaw 전용, Hermes Agent 직접 호환 X

## 3. qmd (tobi) — ⭐27,192

**URL:** https://github.com/tobi/qmd
**Stack:** TypeScript, Node.js
**설치:** `npm install -g qmd` 또는 `npx qmd`

### 핵심 특징
- **CLI search engine** for markdown files
- **Hybrid BM25 + Vector search** — keyword + semantic 동시
- **LLM re-ranking** — 1차 검색 결과를 LLM이 재정렬
- **MCP server 내장** — LLM이 native tool로 사용 가능
- **완전 로컬** — 외부 API 불필요
- **빠름** — Rust 기반? (TypeScript지만 빠름)

### 우리에 적용할 아이디어
- BM25 fallback 전략 (embedding 없이도 검색 가능)
- LLM re-ranking pipeline
- MCP server 아키텍처 (skill plugin 대신 MCP로 노출)

### Karpathy의 평가 (원문 gist에서):
> "qmd is a good option: it's a local search engine for markdown files with hybrid BM25/vector search and LLM re-ranking, all on-device. It has both a CLI (so the LLM can shell out to it) and an MCP server."

## 4. knowledge-manager (treylom) — ⭐177

**URL:** https://github.com/treylom/knowledge-manager
**Stack:** Claude Code plugin, Zettelkasten
**설치:** Claude Code 플러그인 `/plugin install knowledge-manager`

### 핵심 특징
- **Claude Code 전용** — Claude Code CLI에서 `/km:setup` → 웹/PDF/YouTube/카톡 수집
- **Zettelkasten 원칙** — 원자적 아이디어 추출, 3-tier 구조화
- **다중 출력:** Obsidian / Notion / 로컬 Markdown
- **한국어 완벽 지원** — 카카오톡 수집 + 전송
- **모바일 지원** — Claude Code Remote Control에서 keyword-based preset 실행
- **PPT/슬라이드 생성** — AI 이미지 기반, 15+ 스타일

### 우리 환경과의 차이
- Claude Code 전용 — Hermes Agent와 다른 플랫폼
- Zettelkasten 구조 — 우리의 Karpathy Wiki + frontmatter 구조와 다름
- 직접 사용 불가, 아이디어만 참고

## 5. Karpathy LLM Wiki (원본 gist)

**URL:** https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f

### 3-Layer Architecture
1. **Raw sources** — 불변 원본 문서 (articles, papers, transcripts)
2. **The wiki** — LLM이 생성/유지보수하는 markdown files (entities, concepts, comparisons)
3. **The schema** — LLM의 wiki 유지보수 규칙 (예: CLAUDE.md, AGENTS.md)

### 핵심 통찰
- RAG는 매 질문마다 지식을 **재발견**. Wiki는 한 번 컴파일하고 **유지보수**.
- LLM은 북키핑(15개 파일 동시 업데이트, cross-reference 유지)에 강함.
- Human = curation / direction. LLM = maintenance.

### 우리와의 차이
- 원본은 단일 `~/wiki/` 디렉토리. 우리는 multi-repo + submodule 구조.
- 원본은 검색 엔진 없음 (qmd를 optional로 추천). 우리는 Neo4j + Vector Index 구축 예정.
- 우리는 frontmatter 기반 Graph RAG Lite (tags + related)를 이미 운영 중.
